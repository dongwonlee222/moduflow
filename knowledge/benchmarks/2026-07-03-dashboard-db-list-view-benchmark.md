# Dashboard DB/List View Benchmark

## Question

How should ModuFlow improve `memory/dashboard.html` so it remains useful as an issue graph, but also supports PM-style scanning, filtering, and triage?

## Current ModuFlow Gap

The current dashboard is strong at relationship inspection:

- issue graph
- memory graph
- issue drill-down pages
- linked memory previews

But it is weak at operational scanning:

- no dense list of all issues
- no sortable status/phase/next-command table
- no quick "what needs attention" filter
- no Notion/Jira/Linear-style database view over the Git issue files

## Benchmarked Patterns

### Notion

Notion treats one database as a shared data model with multiple saved views. Its documented view types include table, board, timeline, calendar, list, gallery, and chart. Each view can have its own layout, property visibility, filters, sorts, groups, sub-groups, search, and page-open behavior.

Relevant pattern for ModuFlow:

- one canonical issue dataset
- multiple human views over the same rows
- table/list for scanning
- board for status movement
- timeline for dates
- page side peek equivalent via issue drill-down panel

### Jira

Jira positions project work around planning, tracking, dependencies, reporting, and multiple views. Atlassian describes progress monitoring through boards, lists, timeline, or calendars, and highlights dependency management and status/risk updates as core project-management jobs.

Relevant pattern for ModuFlow:

- dashboard should answer "track status" before "show topology"
- list/table and timeline are first-class views, not secondary exports
- dependencies still matter, but should complement a scannable work list

### Linear

Linear custom views are durable filtered views of issues, projects, or initiatives. Users can save, share, favorite, and open filtered boards or lists. Linear also supports view sidebars, quick filters, group header collapse in list view, and project views for "at risk", quarterly review, projects by user, upcoming releases, and projects by initiative.

Relevant pattern for ModuFlow:

- filtered views should be durable or linkable through URL hash/query state
- "at risk" and "review-ready" views matter more than generic raw lists
- list and board should share the same filter model
- side panel/drill-down should clarify a selected item without losing the list

## Recommended ModuFlow Shape

Add a dashboard workbench with these tabs:

1. `이슈 그래프`: current relationship graph.
2. `이슈 DB`: dense table/list view.
3. `칸반`: grouped by status or phase.
4. `타임라인`: optional, date-based view when enough date metadata exists.
5. `지식 그래프`: current memory graph.

V1 should implement `이슈 DB` first, because it has the highest scanning value and lowest implementation risk.

## Proposed Columns For `이슈 DB`

- Issue id
- Title
- Status
- Phase
- Goal
- Next command
- Artifact coverage: issue/spec/plan/tasks/status/review/pr
- Linked memory count
- Relationship count
- Updated date
- Attention flags:
  - no review
  - no PR handoff
  - no Korean sidecar
  - stale lifecycle
  - blocked/no next command

## V1 Interaction Model

- Static HTML generated by `product:dashboard`; no backend.
- Search by issue id/title/text.
- Filter chips for `backlog`, `active`, `review`, `done`, `blocked`.
- Sort by issue id, updated date, status, linked memory count.
- Row click opens `memory/issue-<id>.html`.
- Preserve graph tab and graph node click behavior.

## Decision

Create Issue 056 to add a database/list-style dashboard view before attempting heavier board/timeline editing. Use the benchmark as input to the spec.
