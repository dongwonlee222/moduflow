# Spec: Memory Capture And Sync Workflow

Issue: `034-memory-capture-and-sync-workflow`

## Problem

ModuFlow has a portable project memory prototype, but the user-facing workflow is not yet clear enough:

- Users cannot easily tell when memory is saved.
- Capture is mostly manual, with weak workflow-generated suggestions.
- Full deliverables and durable memory summaries can blur together.
- Downloaded ModuFlow users need a local-only path that works without Dongwon's environment.
- Google Drive or similar storage should be possible, but must not replace Git-tracked Markdown as the source of truth.

## Goals

- Make memory capture understandable: what is saved, when it is saved, and who approves it.
- Separate deliverables from memory entries.
- Add reviewable memory candidates before durable storage.
- Define retrieval behavior for search, evidence review, and session-start context.
- Define external mirror/export adapters for Google Drive, GitHub Wiki, Obsidian folders, or rclone-backed cloud storage.
- Keep local Markdown files canonical so downloaded users can use the feature without hosted infrastructure.

## Non-Goals

- Capture every chat message automatically.
- Store canonical memory only in Google Drive, vector databases, hosted APIs, or MCP indexes.
- Import private personal memory into project repos automatically.
- Build a hosted sync product in this issue.
- Replace issues, specs, roadmaps, or deliverable artifacts.

## Concepts

### Deliverable

A full artifact produced by project work, such as a spec, plan, report, market-entry analysis, benchmark, release note, or exported document.

Deliverables stay in their native project folders:

- `specs/`
- `business/`
- `reports/`
- `workspace/`
- `workflow/`
- other project-local artifact folders

### Memory Entry

A compact durable record that explains why something matters later. It links to source artifacts instead of duplicating them.

Memory entries live under `memory/<kind>/` and include:

- title
- kind
- summary
- source event
- source artifact links
- issue/spec links
- tags
- rationale
- evidence
- owner
- confidence
- review or reversal conditions
- supersedes/superseded_by when applicable
- optional mirror/export metadata

### Memory Candidate

A proposed memory entry generated by a workflow event but not yet accepted into durable memory.

Candidates should be reviewable and editable before becoming permanent memory.

## Capture Triggers

### Manual

The user explicitly asks:

- "이 결정 장기기억에 저장해줘"
- "이 산출물 memory에 등록해줘"
- `product:memory --kind decision --title ...`

Manual capture writes directly when enough fields are supplied. If fields are missing, ModuFlow should ask for the missing high-value fields or create a candidate.

### Suggested

The agent notices a durable event and suggests capture:

- a product or technical decision is made
- a report or business document is completed
- evidence is gathered for an issue
- a release or handoff is completed
- a failed approach should not be repeated

Suggested capture should not silently persist. It creates a candidate or asks for confirmation.

### Workflow-Generated

ModuFlow workflows create memory candidates at predictable lifecycle moments:

- `product:spec` complete -> deliverable candidate
- `product:plan` complete -> deliverable candidate
- `moduflow:business-plan` complete -> deliverable/evidence/decision candidates
- `product:review` complete -> evidence or note candidate
- `product:release` complete -> release candidate
- guarded `done` complete -> summary candidate

## Candidate Lifecycle

```text
detected event
  -> memory candidate
  -> user/agent review
  -> approved memory entry
  -> searchable memory
  -> stale/superseded review when source changes
```

Candidate files can live in a project-local staging folder such as:

```text
memory/.candidates/
```

Approved files move to:

```text
memory/<kind>/<date-slug>.md
```

Rejected candidates should be deleted or archived only with explicit user intent.

## Retrieval Flow

### Search

`product:memory --search <query>` should search canonical memory entries by:

- query text
- kind
- issue ID
- spec path
- artifact path
- tag
- owner
- confidence

Results should explain why they matched and link to the source artifact.

### Evidence Review

`product:evidence` should read memory entries first, then source artifacts as needed:

```text
memory search -> ranked memory entries -> source artifact links -> evidence summary
```

### Session Brief

A future session-start brief can summarize:

- active goal
- active issue
- recent decisions
- recent deliverables
- unresolved memory candidates
- stale or superseded memories
- known failed approaches

## External Mirror And Export Policy

Canonical source:

```text
repo-local Markdown in memory/
```

Mirrors and adapters:

- Google Drive folder mirror
- GitHub Wiki export
- Obsidian vault mirror
- rclone-backed cloud sync
- optional SQLite/vector index
- optional MCP server

Rules:

- Mirrors are rebuildable from repo-local memory and deliverables.
- External indexes may improve search but do not own truth.
- Drive sync should avoid putting `.git/` itself under unreliable cloud conflict handling.
- Exported memory should use relative links where possible.
- Sensitive/private personal memory is never imported automatically.

## Open-Source Pattern Review

- projectmem: strong model for typed events, session briefs, stale-memory detection, and pre-action warnings.
- Basic Memory: strong model for Markdown-first knowledge that humans and LLMs can both edit.
- mem0: strong model for simple add/search API ergonomics.
- Supermemory: strong model for connectors and hybrid search; useful as adapter inspiration, not as the default canonical store.

## Acceptance Criteria

- Downloaded ModuFlow users can initialize, capture, search, and retrieve memory without hosted services.
- Workflows generate reviewable candidates rather than silently saving everything.
- Memory entries link to deliverables instead of duplicating full deliverables.
- Search results include match reason and source links.
- Google Drive and similar tools are documented as mirror/export options.
- Validation detects missing memory structure and broken memory links.
