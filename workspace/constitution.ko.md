# ModuFlow 프로젝트 헌법

버전: **v1.0** (2026-07-07 비준 — 개정 로그 참조)
관할: 이 프로젝트의 엔지니어링 원칙. 에이전트 출력/형식 관례는 `AGENTS.md` 소관(이슈 060 경계) — 둘은 겹치지 않는다.

## 개정 절차

1. 누구든(인간·에이전트) 개정을 **제안**할 수 있다: 이 파일의 diff + 근거.
2. **채택은 명시적 인간 승인 필요**, 개정 로그에 기록(날짜·변경·제안자·인간 승인자). v1.0 비준 자체에도 적용. 에이전트는 승인자 칸을 스스로 채우지 않는다.
3. 버전 범프: **major** — 원칙 추가/삭제 또는 강도(MUST/SHOULD) 변경; **minor** — 의무 변화 없는 표현 명확화.
4. **미기록 편집은 무효**: 승인된 로그 항목 없이 원칙을 바꾼 변경은 효력이 없다. 되돌림 경로: `git log -- workspace/constitution.md`에서 인간 승인 로그 없는 커밋을 revert.
5. 플랜이 한 이슈에 한해 원칙을 **강화**하는 것은 플랜 고유 *추가분*이다 — 개정이 아니며 로그 불필요.

## 원칙

각 원칙: 강도 · 규칙 · 근거 · 출처 (실행된 법의 성문화 — 이 문서를 위해 발명된 것은 없다).

- **C1 · MUST — 주입 가능한 runner.** 스크립트의 모든 git/서브프로세스 접근은 주입된 커맨드 runner를 거친다. *근거: 라이브 프로세스 없는 테스트 가능성; 모든 게이트가 FakeRunner로 검증 가능.* 출처: 075 A1 / `project_sync.py` 패턴, `tests/` 전반.
- **C2 · MUST — 게이트 경로에 조용한 예외 금지.** 게이트/검사의 실패는 명시적 에러 결과로 표면화; `except: pass`와 에러 시 빈 결과 반환 금지. *근거: 075 이전 `release_check` 구멍이 CI에서 공허 통과했다.* 출처: 075 GC2, 수리된 `release_check.py`.
- **C3 · MUST — 동작 변경엔 집중 테스트 동반.** 동작이 바뀌면 TDD; 모든 코드 태스크는 에러 경로 포함 집중 unittest와 함께 착지. *근거: 439개 스위트가 프로젝트의 회귀 바닥.* 출처: superpowers 실행 브리지; 007→072 전 플랜.
- **C4 · MUST — 세션 훅은 fail-open.** 훅 스크립트는 항상 exit 0, 하드 시간 예산 준수, 진단은 파일 로그만; 차단 채널(Stop exit 2, `decision:"block"`) 금지. *근거: 훅 실패가 세션을 희생시켜선 안 된다.* 출처: 072 스펙 + `hook-schema-notes.md`.
- **C5 · MUST — 발견은 append-only.** 자동 패스(converge·보존·sync)는 append만; 기존 산출물 내용의 재작성·재번호·재정렬·삭제 금지, no-op 실행은 파일을 byte 단위로 불변으로 두며 빈 헤더를 만들지 않는다. *근거: 이력이 곧 증거; spec-kit 가드레일 원문 채택.* 출처: 071 가드레일.
- **C6 · MUST — canonical 전환은 인간 게이트.** main 병합, 릴리즈, no-issue 선언, 레코드→이슈 승격 채택, 헌법 개정은 명시적 인간 승인 필요; 에이전트는 제안, 인간이 채택. *근거: 에이전트가 자기 승인을 세탁할 수 없어야 한다.* 출처: 075(신원 게이트 선언), PR #7–#10 실천.
- **C7 · SHOULD — 모델 티어 디스패치.** 판단(범위·저작·종합·사용자 소통)은 메인 루프에; 기계적 구현·조사·독립 검증은 디스패치마다 티어를 명시해 저렴한 티어로. *근거: 비싼 티어는 필요한 곳에; 검증은 구현과 독립.* 출처: 067 관례(2026-07-05), 075/071/072 웨이브 실천.
- **C8 · MUST — 산출물당 파서 하나.** 각 산출물 형식은 정확히 한 곳에서 파싱; 모든 소비자는 그 출력을 쓴다. *근거: 이중 검증기는 갈라진다(OpenSpec #498).* 출처: 071 GC1; `linkage_check`/`project_converge` 실천.
- **C9 · SHOULD — 새 인간 대상 산출물엔 한국어 사이드카.** 새 스펙/패킷은 `.ko.md` 동반; 영어가 canonical; 사이드카 부재는 폴백일 뿐 게이트 아님. *근거: 한국어 사용자의 첫 리뷰 표면.* 출처: 049 관례.
- **C10 · MUST — 동작 변경은 커밋↔이슈 연결.** behavior 영향 작업은 `codex/<issue-id>*` 브랜치 또는 `Issue: <id>` 트레일러; 릴리즈 게이트가 강제, Stop 훅이 세션 중 경고. *근거: 074의 조용한 임계 통과는 다시 불가능해야 한다.* 출처: 075 연결 규약 + 072 경고.
- **C11 · SHOULD — 조용한 캡 금지.** 도구 출력의 절단·샘플링·상한은 스스로 보고한다(`truncated` 필드, 누락 수 표시). *근거: 조용한 절단은 전체 커버리지로 읽힌다.* 출처: 071 증거 번들; 075 리뷰 실천.
